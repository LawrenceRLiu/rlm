## Interesting bumps (in order of importance)

### B1. The workspace Docker image has no `pytest` (3b, turn 2)

**Symptom:** model emitted `<action tool="shell">pytest test_sort.py</action>`; container responded `bash: pytest: command not found` (exit 127).

**Why it matters:** anyone whose task asks the model to "run tests" will hit this. The model recovered (3b, turn 3) by `pip install pytest` inside the running container â€” but that costs a turn and ~6 s, and the install isn't persisted across runs since each run gets a fresh container/workspace.

**Where to inspect:** turn 2 of `_setup_runs/logs/rlm_2026-05-09_17-12-59_48e29be2.jsonl`. The script ran was `_rlm_state/_tmp/shell_t2.a1.sh` â€” visible in the workspace's git snapshot for that turn.

**Suggested fix:** add `pytest` to `docker/workspace.Dockerfile`'s pip-install list (alongside numpy/scipy/etc.). It's tiny and fixes a class of "run my tests" tasks.

### B2. Substrate "halt mutating-after-failure" rule didn't fire on shell exitâ‰ 0 (3b)

The substrate's documented rule is "mutating tool failures halt the rest of the batch." `shell` is mutating. But in 3b turn 2, `shell` returned with `error: 'shell exited with code 127'` AND `stderr: pytest: command not found`. There were no subsequent actions in that turn (the model only emitted the one `shell`), so we didn't actually exercise the halt path here. Worth a deliberate test: emit `<action tool="shell">false</action>` then `<action tool="write_file" path="x">x</action>` in one turn and confirm the write doesn't run.

**Where to inspect:** `_setup_runs/logs/rlm_2026-05-09_17-12-59_48e29be2.jsonl`, turn 2.

### B3. Container writes files as root:root, blocking user-side cleanup

Files written from inside the container land owned by `root:root` (it runs as root by default). With `cleanup_mode="keep"`, removing kept workspaces later requires `sudo`.

**Where to inspect:** `ls -la ~/.rlm/workspaces/run_*` â€” `primes.txt`, `sort.py`, `collated.txt`, etc. all `root:root`.

**Suggested fix:** start the container with `--user $(id -u):$(id -g)` in `rlm/environments/docker_workspace.py:222-256`. Probably need to also chown `/workspace` to the user inside the image, or mount with the right ownership.

### B4. Per-child trajectories aren't in the parent JSONL (3c)

The parent's observation for each `rlm_query` has the `artifacts:` list and a stdout summary, but the `rlm_calls` field is empty (`[]`). The child's per-turn actions/observations don't show up in the parent log â€” they exist only in the child's stdout summary. The visualizer can't drill into a child's reasoning.

**Where to inspect:** `_setup_runs/logs/rlm_2026-05-09_17-14-57_5d23f2bb.jsonl`, turn 2, look at `observations[0..4].rlm_calls` â€” all empty arrays.

**Suggested fix:** either populate `rlm_calls` in the parent observation with the child iteration list, OR write each child to its own JSONL file under `log_dir/children/`. The visualizer's TS types already define `rlm_calls`, so the schema is ready â€” it's a producer-side gap.

### B5. No observation spills hit, even on the 50 KB long-context task (3d)

The 16 KB observation cap (`observation.max_observation_chars`) never tripped because the model used `python` to keep the corpus on disk and only printed `Total Modules: 42 / Total Functions: 420` (small). This means the spill path (`_rlm_artifacts/_observations/`) is **untested by these runs**. Worth a deliberate task that forces a big observation â€” e.g. `<action tool="shell">cat _rlm_query_0.txt</action>` on a 50 KB file would spill.

### B6. Initial bootstrap failure due to model_name vs alias confusion

The first run tried `model_name="google/gemma-4-31B-it"` and got `400 Invalid model name passed in model=google/gemma-4-31B-it. Call /v1/models to view available models.` Litellm requires the alias from `model_list[*].model_name` (`gemma-4-31b` here). This is a one-line trap that's now in `../SETUP.md` but worth flagging again.

**Where to inspect:** the failed JSONL is `_setup_runs/logs/rlm_2026-05-09_17-11-52_aeee5315.jsonl` â€” has the metadata header but no iterations because the first LM call 400'd.

## Substrate health snapshot (4 runs)

- **Parse retries: 0 across all 4 runs.** Gemma 4 31B follows the `<action>` XML format cleanly. Worth comparing against Qwen3.6 (per `Todo.md`, failed format on the old REPL substrate).
- **Observation spills: 0.** Spill path is untested by these runs (see B5).
- **Tool errors: 1 total** (3b's pytest 127). Recovered.
- **Recursion artifact handoff: clean.** 5/5 children's artifacts landed correctly with the path-mapping observation.
- **Substrate didn't crash, hang, or produce malformed observations on any turn.**

## What this tells us about substrate vs. model

The substrate is solid for everything I exercised â€” including the recursion path that the migration docs flag as the most fragile. The bumps were either model-planning quirks (3b's run-then-write order) or environment gaps (no `pytest` in the workspace image, root-owned files, missing child trajectory data in parent logs). None of those are deal-breakers; they're each a small, isolated fix.
