## What Qwen3.6 actually emits ‚Äî a concrete example

From 3c turn 1's parse_attempts, here's the model's three failed attempts followed by the eventual success:

**Attempt 0 ‚Äî child element instead of attribute:**
```
The user wants me to start by reading the query file to understand the task. Let me begin by reading `_rlm_query_0.txt`.
</think>

<action tool="read_file"><path>_rlm_query_0.txt</path></action>
```
Parser response: `Missing required attribute(s) ['path'] on <action tool='read_file'>`.

**Attempt 1 ‚Äî drops the `tool` attribute too:**
```
The previous response was flagged as malformed because the parser apparently didn't recognize the `<path>` tag‚Ä¶
```
Parser response: `Missing required attribute 'tool' on <action> element`.

**Attempt 2 ‚Äî same:**
```
The user wants me to read the first query file. I need to make sure I format the action correctly. ‚Ä¶
```
Parser response: `Missing required attribute 'tool' on <action> element`.

**Eventual success (the surviving response that got logged):**
```
The user is asking me to read `_rlm_query_0.txt`. My previous attempts were marked as malformed due to parser issues. Let me try a simpler approach‚Ä¶
</think>

<action tool="read_file" path="_rlm_query_0.txt" />
```

Two things to notice:
1. **Qwen3.6 visibly emits chain-of-thought wrapped in `<think>‚Ä¶</think>` tags** ‚Äî the `</think>` close tag is right above each action. The substrate's parser doesn't care (it just looks for `<action>` blocks anywhere in the response), but it tells you what's pre-`<action>` text in any Qwen3.6 trajectory.
2. **The model's first instinct was XML-with-children**, the second/third instinct was `<action>` with no `tool` attr at all ‚Äî this is a *systematic* schema misunderstanding, not random noise. Even with explicit parser feedback ("Missing required attribute‚Ä¶"), it took until attempt 3 to land on attribute syntax.

For 3b/3d the model never landed it within the 3-retry budget, hence the crashes.

## Issues this run uncovered (new vs. the Gemma trace)

### Q1. Logging gap when a turn exhausts parse retries

When `_call_lm_with_parse_retry` raises `ActionParseError`, the iteration is *not* recorded in the JSONL ‚Äî only the metadata header survives. So 3b's run has `iterations logged: 0` despite three real LM calls happening on turn 1. The `parse_attempts` for the failed turn are lost.

**Why it matters:** post-mortem for a crashed run is much harder. Right now you have to grep the conda env's stderr or re-run with extra logging. The whole point of `parse_attempts` is to *see* the model's malformed responses, but they're only persisted on turns that eventually succeed.

**Where:** `rlm/core/rlm.py:333` ‚Äî the iteration is built *after* `_call_lm_with_parse_retry` returns. If the call raises, no iteration is appended to the logger. Suggested fix: catch `ActionParseError` in `_completion_turn`, build a partial iteration with the parse_attempts and the error, log it, then re-raise.

### Q2. Children inherit the same parse-retry budget but not the parent's hard-won format knowledge

In 3c, the parent took 3 retries on turn 1 to discover the right `<action tool="..." path="..."/>` syntax. Then it spawned 5 children. **Each child started fresh** with no carryover ‚Äî they each had to rediscover the same syntax. None did.

**Why it matters:** for any model that needs the parser feedback loop to learn the schema, recursion will fail catastrophically because each new child is a cold start. Effectively `max_concurrent_subcalls` is 0 for non-format-following models.

**Where:** `rlm/core/recursion.py` spawns children with the same system prompt as the parent. There's no mechanism to carry over "successful action examples" or to seed the child with a one-shot from the parent's recovered turn. Suggested investigation: add a few-shot example to the system prompt drawn from the parent's last successful turn.

### Q3. Qwen3.6's thinking-mode `<think>` tags are noise the parser handles, but they bloat token counts

The model emits long `<think>‚Ä¶</think>` reasoning blocks before each action. The substrate's parser ignores them (it scans for `<action>` regardless of surrounding text), so this isn't a bug ‚Äî but it does mean Qwen3.6 uses ~2√ó the tokens per turn that Gemma 4 does for equivalent work. 3a Qwen used 5,598 in / 830 out vs Gemma 4's 4,076 in / 258 out for the same task with one fewer turn.

**Possible mitigation:** vLLM supports `--reasoning-parser` flags that strip `<think>` blocks server-side. We could enable that for Qwen models specifically. Tradeoff: the parent's prose context may have been useful for the LM's own self-correction (its turn-1 recovery in 3c referenced "previous attempts were marked as malformed", which suggests it's reading parser feedback from earlier turns ‚Äî keeping `<think>` content might help self-debug).

### Q4. Substrate startup quirk: Qwen3.6 has Mamba layers, breaking default `max_num_seqs`

Even on A100 80 GB single-card with 95% memory utilization, vLLM 0.19.1 OOMs during CUDA graph capture for `Qwen/Qwen3.6-35B-A3B` at default `max_num_seqs=256`. The vLLM logs hint at it: `Capping cudagraph capture sizes from max 512 to 352 to fit Mamba cache blocks (365 blocks available)`. Setting `--max-num-seqs 64` resolves the OOM. This is a Qwen-specific architecture detail (hybrid Mamba-Transformer) that doesn't apply to Gemma 4. Worth pinning in the SETUP doc as a Qwen-specific flag.

## What this tells us about substrate vs. model

The substrate behaved correctly under stress:
- Caught all parse failures, retried per `parse.max_action_parse_retries`, raised the documented exception when exhausted.
- Halted mutating-tool batches on first failure, gave the model a clear "Re-issue this action next turn after addressing the failure" observation.
- Recursion children failures became error observations on the parent, not crashes.
- No memory leaks, no hangs, no malformed observations across 14+ turns of error-heavy traffic.

The model issue is the model issue: Qwen3.6 has not internalized the `<action tool="X" path="Y"/>` attribute schema. The most striking evidence is the parent's eventual success on 3c turn 1 ‚Äî it *can* produce the right format when given enough explicit error feedback, but only after wasting 3 LM calls per attempt. For a substrate that's supposed to scale to recursive tree-of-tool-calls, that overhead is fatal.

Two product directions implied by this run:
1. **Easier-to-follow action format.** This is already in `Todo.md`. The data here makes it concrete: Qwen3.6 specifically wants child elements (`<path>...</path>`), not attributes. Either (a) accept both forms in the parser, or (b) switch the canonical format to child-element style and update Gemma's prompt accordingly.
2. **Few-shot bootstrap for non-format-following models.** Putting one well-formed example in the system prompt (or ‚Äî for recursion ‚Äî passing the parent's last-successful-action as a child's seed) might dramatically reduce the parse-retry burden.
