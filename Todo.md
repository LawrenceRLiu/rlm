## Analysis
Previously we ran a short analysis of RLMs on OOLONG for the more modern models, ie Qwen3.6 and Gemma 4. We found that while Gemma 4 largely can follow the RLM format, the Qwen3.6 models (both 27B dense and 35B-A3B MoE could not). At the same time, the Gemma 4 model was able to follow the RLM format.

# Bumps from second end-to-end run (2026-05-09, Qwen3.6-35B-A3B)
Surfaced by re-running the same four Phase 3 tasks against `Qwen/Qwen3.6-35B-A3B`; documented in `workspace_substrate_arch/dev/2026-05-09_qwen3-6-35B-A3B_vs_gemma4-31B.md`. 1/4 tasks passed (3a primes); 3b/3c/3d all crashed with `ActionParseError`. Two existing test-suite items closed (above). Four new issues:

- [x] **Log a partial iteration when a turn exhausts parse retries.** ~~Currently `_call_lm_with_parse_retry` raises `ActionParseError` *before* the iteration is built~~ — **closed by today's visibility PR.** `_call_lm_with_parse_retry` now attaches `parse_attempts`/`last_response`/`last_reasoning` to the raised exception (`rlm/core/rlm.py:392-397`); `_completion_turn` catches it, builds a partial `WorkspaceIteration` with `error=str(exc)` and the full attempt history, attaches `exc.iteration = partial`, then re-raises (`rlm/core/rlm.py:340-360`). `_run_loop` catches in turn and logs the partial via `self.logger.log_iteration` before re-raising (`rlm/core/rlm.py:267-280`). Verified end-to-end against Qwen 3b: re-run produces 1 iteration line with `actions=0`, `parse_attempts=4`, `error` populated, where the original run had zero iteration lines. New `WorkspaceIteration.error` field added (`rlm/core/types.py:255`); visualizer renders a red `FAILED` badge in `IterationTimeline` and a red error card in `ActionPanel`. (Q1 in Qwen doc.)
- [ ] **Recursion children inherit the parent's parse-retry budget but not its hard-won format knowledge.** In Qwen 3c turn 1, the parent took 3 retries to discover the right `<action tool="..." path="..."/>` syntax. Then it spawned 5 children — every child cold-started and failed. Effectively `max_concurrent_subcalls = 0` for non-format-following models. Investigate carrying over a few-shot example to children (e.g. seed the child's system prompt with the parent's last successful action body), implemented in `rlm/core/recursion.py`. (Q2.)
- [ ] **Optionally enable `--reasoning-parser` server-side for Qwen-family models.** Qwen3.6 emits `<think>…</think>` reasoning blocks before each action. The substrate's parser ignores them so this isn't a correctness bug, but token usage on equivalent tasks is ~2× Gemma 4 (3a Qwen: 5,598 in / 830 out vs Gemma's 4,076 in / 258 out). vLLM supports `--reasoning-parser` to strip these server-side. Tradeoff: removing the prose may hurt the model's self-correction loop, since parser feedback in the next turn references "previous attempts." Worth a careful A/B before enabling globally. (Q3.)
- [ ] **Document Qwen3.6's hybrid-Mamba `--max-num-seqs` requirement in SETUP.** vLLM 0.19.1 OOMs during CUDA-graph capture for `Qwen/Qwen3.6-35B-A3B` at default `max_num_seqs=256` — even on a single A100 80 GB at `--gpu-memory-utilization 0.95`. Mamba state cache competes with KV cache. Setting `--max-num-seqs 64` resolves it. Add a Qwen-specific note to `workspace_substrate_arch/SETUP.md`. (Q4.)

## Two product directions implied by the comparison

- [ ] **Make the action format more forgiving (likely highest-leverage item).** Qwen3.6's systematic mistake is emitting `<action tool="read_file"><path>...</path></action>` (path as child element) instead of `<action tool="read_file" path="..."/>` (path as attribute). Two paths: (a) extend `rlm/utils/action_parser.py` to accept both attribute-form AND child-element-form for path-like args; (b) switch the canonical format to child-element style and update prompts/Gemma examples accordingly. Either would likely flip Qwen3.6 from 1/4 → 4/4. This is a concrete instance of the existing Todo "Make the RLM format more easy to follow for a LLM."
- [ ] **Few-shot bootstrap a well-formed action in the system prompt.** The current system prompt in `rlm/utils/prompts.py` describes the schema but doesn't include a worked example. Adding one well-formed `<action tool="X" path="Y">...</action>` example (and its observation) might dramatically reduce parse-retry burden for non-format-following models without changing the parser at all. Cheap to try; would also benefit recursion children since the example travels with the system prompt.
- [ ] **Investigate prescriptive CoT-prose phrasing in the system prompt.** Gemma-4-31b emits zero prose around its `<action>` blocks across all 81 turns of the SWE-Bench smoke run — the current prompt says "You may write reasoning prose" (permissive), and gemma takes the permissive option to the limit. Trajectory readability suffers and CoT-before-action typically helps agentic perf. A/B the current phrasing vs. "Reason briefly before each action" vs. a dedicated `<reason>` tag, across gemma / Sonnet / Qwen3.6, on TB + SWE-Bench. Details + plan in `workspace_substrate_arch/dev/2026-05-10_cot_prose_investigation.md`.

# Visibility follow-ups (2026-05-09)

Surfaced while implementing the JSONL + visualizer "show child trajectories and failed iterations" pass (today). The XML `<action tool="rlm_query">…</action>` path now plumbs the full child trajectory into the parent observation's `rlm_calls`, which the visualizer renders with a recursive drill-down. There is one analogous hole left:

- [ ] **`rlm_query_batched` / `llm_query_batched` from inside `python` actions are invisible in the JSONL.** When a `python` script calls `rlm_query_batched(["t1", "t2", ...])`, the host's `RecursionHandler.spawn_via_broker_batched` (`rlm/core/recursion.py:98-116`) actually produces a full `WorkspaceObservation` (with populated `rlm_calls=[child_RLMChatCompletion]`) for each child — and then throws all of it away, returning only `{"responses": [obs.stdout, ...]}` to the broker. The strings flow back into the script's stdout; the `python` tool wraps that into a `WorkspaceObservation` with `rlm_calls=[]`. So a fan-out via `rlm_query_batched` (the natural pattern for tasks like 3c) is opaque to the visualizer even though the child trajectories existed in memory. Same gap exists for `llm_query` / `llm_query_batched` from inside `python` (`rlm/environments/docker_workspace.py:322-343`).

    *Note: this is strictly worse than the XML `rlm_query` case the rest of today's work fixed — XML produces one observation per child (1:1 with `rlm_calls`), python-batched collapses N children into one tool's stdout.*

    Sketch of the fix:
    1. In `recursion.py:98-116` and the two `_do_*_batched` handlers in `docker_workspace.py:331-360`, keep the per-child `RLMChatCompletion`s alongside the response strings.
    2. Add a thread-safe per-action ledger on `DockerWorkspaceEnv` keyed by `current_action_id` (broker requests run in worker threads while `exec_in_container` blocks on the python action). Every `_handle_broker_request` worker appends its produced `RLMChatCompletion`s to the ledger entry for the currently-executing action.
    3. In `rlm/workspace_tools/python.py:67-79`, after `exec_in_container` returns, drain the ledger for this `action_id` into `obs.rlm_calls` (and clear the entry).
    4. New integration test in `tests/test_docker_workspace.py`: a python action that calls `rlm_query_batched(["a","b"])`; assert the resulting observation has `len(obs.rlm_calls) == 2` with each carrying populated `metadata.iterations`.

    No visualizer work needed — `ActionCard`'s "Sub-LM Calls" section already maps over `observation.rlm_calls` and shows the drill-down button on any whose `metadata.iterations` is non-empty.

    Trickiest bit is the ledger keying. Since `current_action_id` is mutated by the parent loop as actions are dispatched, and broker requests are processed in worker threads, snapshot the action_id at the start of `_handle_broker_request` rather than reading it lazily.
